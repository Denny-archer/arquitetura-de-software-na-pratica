# Arquitetura de Software na Prática

## Sobre o Projeto

Este projeto foi desenvolvido como parte do desafio da DIO utilizando o NotebookLM como ferramenta de aprendizagem ativa.

O objetivo foi estudar os principais conceitos de Arquitetura de Software, compreender diferenças entre arquiteturas monolíticas e microsserviços, analisar princípios de design e consolidar o conhecimento em um guia de estudos reutilizável.

---

## Objetivos de Estudo

* Compreender os fundamentos da Arquitetura de Software.
* Entender as diferenças entre Monólitos e Microsserviços.
* Estudar Clean Architecture e princípios SOLID.
* Conhecer padrões de projeto voltados para sistemas escaláveis.
* Desenvolver senso crítico para escolha arquitetural em projetos reais.

## Miniguia de Estudos

### O que é Arquitetura de Software?

Arquitetura de Software é a estrutura fundamental de um sistema, definindo componentes, responsabilidades, interações e decisões técnicas que influenciam qualidade, escalabilidade e manutenção.

---

### Arquitetura Monolítica

Características:

* Aplicação única
* Implantação simplificada
* Menor complexidade inicial

Desvantagens:

* Escalabilidade limitada
* Maior acoplamento

---

### Arquitetura de Microsserviços

Características:

* Serviços independentes
* Deploy isolado
* Escalabilidade individual

Desvantagens:

* Complexidade operacional
* Monitoramento mais difícil

---

### Clean Architecture

Princípios:

* Independência de frameworks
* Separação de responsabilidades
* Regras de negócio isoladas

Benefícios:

* Facilidade de manutenção
* Melhor testabilidade
* Maior flexibilidade tecnológica

---

### SOLID

* S — Single Responsibility Principle
* O — Open/Closed Principle
* L — Liskov Substitution Principle
* I — Interface Segregation Principle
* D — Dependency Inversion Principle

---

## Glossário

| Conceito           | Definição                                                  |
| ------------------ | ---------------------------------------------------------- |
| Monólito           | Aplicação construída como uma única unidade                |
| Microsserviço      | Serviço independente responsável por uma função específica |
| Escalabilidade     | Capacidade de crescer mantendo desempenho                  |
| Acoplamento        | Grau de dependência entre componentes                      |
| Coesão             | Relacionamento interno de responsabilidades                |
| SOLID              | Conjunto de princípios para design orientado a objetos     |
| Clean Architecture | Modelo arquitetural focado em independência e manutenção   |

---

## Principais Aprendizados

Durante este estudo foi possível perceber que não existe uma arquitetura universalmente melhor. A escolha entre monólitos e microsserviços depende do contexto do negócio, maturidade da equipe, complexidade do sistema e requisitos não funcionais.

A principal conclusão foi que decisões arquiteturais devem ser guiadas por necessidades reais do projeto e não apenas por tendências tecnológicas.
